# Как встраивать geolm в другие сервисы

Модель после обучения, это один файл `*.geolm`. Развёртывание, это копирование файла
на нужную машину и выбор одного из трёх способов подключения, описанных ниже.

Важно понимать контракт: модель принимает строку и возвращает строку. Любые данные
(числа, поля формы, JSON) перед подачей приводятся к строке в одном согласованном
формате, точно так же, как договариваются о формате любого API. Одна модель обучена
под одну задачу; как обслуживать несколько видов входа, описано в конце.

---

## Способ 1. HTTP-сервис (для любого языка и любого стека)

Запуск на машине или в контейнере:

```bash
geolm serve --model task.geolm --port 8080
```

Контракт сервиса: `POST /predict`, тело запроса JSON, два варианта.

Одиночный запрос и ответ:

```json
запрос:  {"input": "7 марта 1961"}
ответ:   {"output": "1961-03-07"}
```

Пакетный запрос и ответ (быстрее на поток):

```json
запрос:  {"inputs": ["1+1=", "40+50="]}
ответ:   {"outputs": ["2", "90"]}
```

Примеры клиентов.

curl:

```bash
curl -X POST localhost:8080/predict -d '{"input": "7 марта 1961"}'
```

Python (requests):

```python
import requests
r = requests.post("http://localhost:8080/predict", json={"input": "7 марта 1961"})
print(r.json()["output"])          # 1961-03-07
```

JavaScript (fetch, годится для Node и браузера через ваш бэкенд):

```javascript
const r = await fetch("http://localhost:8080/predict", {
  method: "POST",
  body: JSON.stringify({ input: "7 марта 1961" }),
});
const { output } = await r.json();
```

Go:

```go
resp, _ := http.Post("http://localhost:8080/predict", "application/json",
    strings.NewReader(`{"input": "7 марта 1961"}`))
```

Контейнер (если нужен Docker), минимальный Dockerfile:

```dockerfile
FROM python:3.12-slim
COPY geolm /opt/geolm
RUN pip install --no-cache-dir torch --index-url https://download.pytorch.org/whl/cpu \
 && pip install /opt/geolm
COPY task.geolm /opt/task.geolm
EXPOSE 8080
CMD ["geolm", "serve", "--model", "/opt/task.geolm", "--port", "8080"]
```

## Способ 2. Прямой импорт (если ваш сервис на Python)

Без сети и без отдельного процесса, модель живёт внутри вашего приложения:

```python
from geolm import GeoLM

model = GeoLM.load("task.geolm")          # один раз при старте сервиса

def handle(request_text: str) -> str:      # в обработчике запроса
    return model.predict(request_text)

# для потока запросов быстрее пакетно:
answers = model.predict_batch(["1+1=", "40+50=", "7+8="])
```

Такой способ самый быстрый (нет сетевых накладных расходов) и подходит, например,
для FastAPI и Flask приложений, воркеров очередей, телеграм-ботов.

---

## Способ 3. Конвейер и скрипты (stdin, построчно)

```bash
cat inputs.txt | geolm predict --model task.geolm > outputs.txt
```

Каждая строка входного файла превращается в строку ответа. Удобно для разовой
пакетной обработки и cron-задач.

## Разные виды данных на входе

Правило: модель понимает строки, поэтому «разные данные» решаются на уровне
формата строки. Три типовых случая.

Случай 1, данные структурные (несколько полей). Сериализуйте поля в строку с
разделителем и обучайте модель именно на таком формате:

```json
{"input": "город=Москва; улица=Тверская; дом=1", "output": "119019"}
```

Случай 2, входы разных типов, а модель одна. Добавьте в обучающие данные префикс
задачи, и одна модель научится различать задачи сама (многозадачное обучение):

```json
{"input": "дата: 7 марта 1961",  "output": "1961-03-07"}
{"input": "сумма: 12+34",        "output": "46"}
{"input": "телефон: 8 916 1234567", "output": "+79161234567"}
```

После обучения на смеси модель отвечает по префиксу: `predict("дата: 1 мая 2000")`.

Случай 3, форматов много и они сложные. Держите несколько узких моделей (по файлу
на задачу) и выбирайте модель в коде по типу запроса. Файлы маленькие, и это
нормальная схема: специалист на задачу.

## Правила надёжной эксплуатации

Проверяйте вход до модели. Модель ответит что-нибудь на любой вход, включая мусор,
поэтому валидация формата (длина, допустимые символы) должна стоять перед ней.
Символы, которых не было в обучающих данных, модели неизвестны.

Проверяйте выход после модели. Если ответ обязан быть, например, датой или числом,
проверьте его регулярным выражением и обработайте редкие несоответствия.

Прогревайте модель при старте сервиса одним холостым запросом, тогда первый
настоящий запрос не будет медленнее остальных.

Замеряйте качество на отложенных данных перед каждой выкаткой новой версии модели
(`geolm eval --model new.geolm --data holdout.jsonl`) и не выкатывайте, если
точность упала относительно текущей.

Масштабирование: один процесс держит тысячи запросов в секунду на процессоре;
если нужно больше, запускайте несколько процессов за балансировщиком, модель
не хранит состояния между запросами, поэтому реплики независимы.
